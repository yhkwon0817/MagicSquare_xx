## 3. 일치 확인 (갭 표 제외)

- **Input**: `grid: list[list[int]]` 4×4, 셀 값 0 또는 1~16
- **Command**: `validate_lines(grid)` 단일 호출로 10선 검증 (성공 기준 #1)
- **Output 골격**: `ok`, `status` (`pass`|`fail`|`incomplete`), `lines[]`
- **줄 ID**: `row:0`~`row:3`, `col:0`~`col:3`, `diag:main`, `diag:anti`
- **fail 출력**: 틀린 줄만 `{ id, sum, expected: 34 }`
- **R5**: `0` 포함 → `status=incomplete`, 합 계산·34 비교 생략

## 4. 빠지거나 어긋난 항목 (8건)

| 구분 | 계약 (PRD) | 워크북 (`Report/03`) | 어긋남 |
|------|------------|----------------------|--------|
| **R-G-I-O / O** | R1~R4: 행·열·주대각·부대각 각각 합=34 (§3.2) | O-Output에 **R5(incomplete)만** 명시 | R1~R4 판정 규칙 누락 (§3.5 Rule에만 존재) |
| **R-G-I-O / O** | `pass` → `lines=[]`; `incomplete`(R5) → 검증 생략·`lines=[]` (T1, T3) | `lines[]`를 **fail 시만** 기술 | pass/incomplete 시 `lines[]` 거동 미명시 |
| **R-G-I-O / O** | `ValidationResult.ok` + `status` 쌍 (§5.2, T1~T3) | 타입에 `ok` 있으나 **status별 ok 값** 미정의 | `incomplete`일 때 `ok`=? (AC4: pass 아님) |
| **성공 기준 #2** | AC4: `0` 포함 → `status=incomplete`, **pass 아님** | incomplete Red에 **`status`만**, `ok` 미언급 | incomplete 시 `ok=false`(또는 ok≠true) 누락 |
| **성공 기준 #2** | T4: 대각만 ≠34 → `diag:main`/`diag:anti` (§8) | fail Red를 **합≠34 일반**으로만 기술 | 대각선 단독 실패(T4) 시나리오 누락 |
| **성공 기준 #2** | T5: **여러 줄** 동시 실패 → **모든** 틀린 줄 ID+sum (§8) | fail Red **1케이스** 수준 | 다중 실패(T5) 시나리오 누락 |
| **성공 기준 #2** | AC6·F5: fail Red에서 **실패 줄 ID assert** (§3.5 Test Loop) | `ok`/`status`만, **줄 assert 없음** | fail Red 시 `lines[]`/줄 ID 검증 누락 |
| **성공 기준 #3** | AC3: pass → 10선 모두 sum=34; T3: incomplete → `lines=[]` | **fail 시 `lines[]`** 만 기술 | pass/incomplete의 `lines[]`(빈 배열) 및 AC3 미포함 |

## 5. 결론 및 다음 단계

### 결론

- 워크북 R-G-I-O·성공 기준 3개는 **Mom Test → 10선 판정** 골격과 PRD 계약 **핵심(Output 형태·R5·10선)** 은 일치한다.
- **8건**은 워크북 §3.2·§3.3에 명시되지 않았거나 §3.5 Test Loop·PRD §8과 불일치 — RED 테스트·`docs/PRD.md` 이식 시 보완 대상.

3. **`docs/PRD.md` 초안** — `Report/03` §3 + 본 리뷰 갭 반영 후 `/tdd-red` 진입
